# 蓝牙配网与配置通道（BLE JSON Provisioning）需求与说明

适用工程：`water`  
固件侧模块：`components/blufi_custom`（已移除 BLUFI，保留历史组件名）  
客户端：团队小程序 / App（需按本文档实现 BLE 连接 + Notify + JSON 写入）  

> 本文档描述“首次绑定/重置后配网”的 BLE 通信需求、协议、状态机、时序与边界条件。  
> 设计目标：**简单、稳定、可维护、资源占用更低**（不再依赖 BLUFI 协议栈与其加密/CRC/分包实现）。

---

## 1. 背景与目标

### 1.1 背景
- 传统 BLUFI 在自研小程序端需要完整实现 DH/AES/CRC/分包等协议细节，易出现兼容问题（例如 checksum error）。
- 本项目为家用净水器，配网通道只在**首次绑定或重置后**短时间使用，允许采用更轻量的方案。

### 1.2 目标（Must）
1. 设备支持通过 BLE 在配网窗口内完成：
   - 选择联网方式：Wi‑Fi 或 4G（当前固件支持）
   - Wi‑Fi 配网（SSID/Password）
   - MQTT 服务器配置（Host/Port/Username/Password）
2. 协议与流程对齐《蓝牙通信协议.xlsx》中字段与步骤（并增强“步骤门禁”防乱序）。
3. 设备对客户端提供清晰的状态回包：
   - `statusBLE`、`statusNet`、`statusWiFi`、`statusMQTT`
4. 配网窗口：**每次上电/重启后开放 120 秒**，超时后停止广播并拒绝写入。
5. 单连接：同一时刻只允许 1 个 BLE 连接。

### 1.3 非目标（Nice to have / Not in scope）
- 不提供 BLUFI 兼容（官方 BLUFI App 不再可用）。
- 不提供 BLE 链路加密/配对（明文 JSON）；安全通过“配网窗口 + 步骤门禁”控制。
- 不提供多设备群控/批量配网能力。

---

## 2. 安全模型（约束与防护）

### 2.1 风险假设
该通道为明文 JSON，存在被近距离监听/恶意连接的理论风险。结合产品场景（家用、短窗口）风险可接受。

### 2.2 防护策略（已实现）
1. **配网窗口**：仅在上电/重启后 120 秒内允许写入；超时后：
   - 停止广播
   - 断开当前 BLE 连接（如存在）
   - 后续写入返回“写入不允许”
2. **步骤门禁**：必须按步骤顺序发送 JSON；不符合当前步骤的消息将被忽略（并在串口打印 step mismatch）。
3. **单连接**：减少多端并发导致的状态串扰。

> 可选增强（未来）：增加 session token/nonce，要求后续每个包携带 token 才接受。

## 3. BLE 协议概览

### 3.1 广播与设备名
- 广播名（Device Name）：`WATER_PROV`
- 广播时机：配网窗口打开后立即广播；窗口关闭后停止广播。

### 3.2 GATT 服务与特征值（128-bit UUID）

小程序需发现以下 UUID，并按用途订阅/写入：

| 类型 | 名称 | UUID | 属性 | 用途 |
|---|---|---|---|---|
| Service | Provision Service | `9E3C0001-2C6B-4F9B-8B5A-6F686E3D110A` | Primary | 自定义配网服务 |
| Characteristic | RX | `9E3C0002-2C6B-4F9B-8B5A-6F686E3D110A` | Write / WriteNoRsp | 小程序写入 JSON 到设备 |
| Characteristic | TX | `9E3C0003-2C6B-4F9B-8B5A-6F686E3D110A` | Notify | 设备通知回包 JSON 给小程序 |

### 3.3 必要的客户端行为（强制）
1. 连接成功后，**必须订阅 TX 的 Notify**（写 CCCD 0x0001）。
2. 建议请求更大 MTU（例如 256），但不能依赖一定成功：
   - 最终生效 MTU = Min(手机支持最大值, ESP32 支持最大值)。
   - iOS 端对 MTU 协商更严格，`wx.setBLEMTU` 可能无效；强行写长包可能导致 `writeBLECharacteristicValue` 失败（如 10008）。
3. 必须支持“应用层分包/重组”（见 3.4），用于：
   - 小程序 -> 设备：长 JSON 写入
   - 设备 -> 小程序：长 JSON Notify 回包
4. 写入建议使用 **Write（带响应）**或 WriteNoRsp 均可；调试阶段推荐 Write（带响应）。

### 3.4 应用层分包协议（PV1，强制支持）

为兼容“老手机 MTU 仅 23”以及 iOS 端 MTU 协商不稳定的情况，本项目在 RX/TX 两条通道均支持 PV1 分包帧。

#### 3.4.1 何时需要分包
- 当单条 JSON 长度 `> (MTU - 3)` 时，必须分包，否则会写入失败或被底层截断。
- 推荐策略：小程序侧无论 MTU 是否成功协商，均按以下规则发送：
  - `len <= (MTU - 3)`：可直接写 JSON（以 `{` 开头的文本）
  - `len > (MTU - 3)`：使用 PV1 分包帧发送

#### 3.4.2 帧格式
- Header 固定 8 字节，小端序：
  - Byte0：`0x50`（'P'）
  - Byte1：`0x56`（'V'）
  - Byte2：`0x01`（版本）
  - Byte3：`flags`（bit0=START，bit1=END）
  - Byte4-5：`total_len`（本条 JSON 总长度，仅 START 帧必须正确）
  - Byte6-7：`offset`（当前 payload 在整条 JSON 中的偏移）
  - Byte8..：`payload`（连续的 JSON 字节切片）

#### 3.4.3 分包大小
- 单帧最大有效载荷：`payload_max = (MTU - 3) - 8`
- 当 MTU=23 时：`payload_max = 12`，可稳定兼容老设备

#### 3.4.4 重组规则
- 设备侧要求：分包必须按 `offset` 从 0 连续递增拼接；中途丢包/乱序会导致本条消息丢弃。
- 小程序侧在接收 TX Notify 分包时，也需按同样规则重组；重组完成后再对 JSON 做解析。

## 4. JSON 消息格式约定

### 4.1 编码
- 编码：UTF-8
- 数据类型：文本 JSON（对象）
- 空格/换行：允许（固件侧按 json_parser 解析）
- 解析：固件侧使用 `json_parser` 解析 JSON


### 4.2 字段冲突约定（重要）
- Wi‑Fi 配置消息通过 `ssid` 字段识别；**仅出现 `password` 不代表 Wi‑Fi**（因为 MQTT 也有 `password`）。

---

## 5. 流程与状态机（步骤门禁）

### 5.1 步骤定义

| Step | 状态机名 | 小程序 -> 设备 | 设备 -> 小程序 | 说明 |
|---|---|---|---|---|
| 1 | WAIT_BLE_OPEN | `{"statusBLE":0}` | `{"statusBLE":0}` | 握手开始 |
| 2 | WAIT_NET_MODE | `{"statusNet":0/1}` | `{"statusNet":1}` | 选择 Wi‑Fi(0) / 4G(1)，设备返回确认 |
| 3（Wi‑Fi） | WAIT_WIFI_CFG | `{"ssid":"...","password":"..."}` | `{"statusWiFi":0}` | 设备开始连接 Wi‑Fi 时回 0 |
| 3（Wi‑Fi） | WAIT_WIFI_RESULT | （无） | `{"statusWiFi":1/2}` | 拿到 IP 回 1；最终失败回 2 |
| 4 | WAIT_MQTT_CFG | `{"mqttserver":"ip:port","username":"...","password":"..."}` | `{"statusMQTT":0/1}` | 设备等待 MQTT 成功/失败回包 |
| 5 | WAIT_BLE_CLOSE | `{"statusBLE":1}` | `{"statusBLE":1}` | 结束流程；设备关闭窗口并断开 BLE |

### 5.2 Wi‑Fi 回包定义（对齐文档）
- `{"statusWiFi":0}`：开始连接
- `{"statusWiFi":1}`：成功（拿到 IP）
- `{"statusWiFi":2}`：失败（含重试耗尽或密码错误）

### 5.3 MQTT 回包定义（对齐文档 + 当前固件判定）
- `{"statusMQTT":0}`：连接成功  
  **当前固件判定条件**：收到 `APP_EVENT_MQTT_PLAN_RECEIVED`（代表已连上并收到套餐/plan 信息）。
- `{"statusMQTT":1}`：连接失败或超时  
  - 超时默认 20 秒（可配置）
  - 或收到 `APP_EVENT_MQTT_DISCONNECTED`

---

## 6. 时序（推荐实现）

### 6.1 Wi‑Fi 模式完整时序（建议）
1. 连接 BLE -> 订阅 TX Notify -> 请求 MTU=256
2. 发送 `{"statusBLE":0}`，等待回 `{"statusBLE":0}`
3. 发送 `{"statusNet":0}`，等待回 `{"statusNet":1}`
4. 发送 `{"ssid":"...","password":"..."}`，等待回 `{"statusWiFi":0}`
5. 等待设备回 `{"statusWiFi":1}`（成功）或 `{"statusWiFi":2}`（失败）
6. 发送 MQTT 配置 JSON，等待回 `{"statusMQTT":0/1}`
7. 发送 `{"statusBLE":1}`，等待回 `{"statusBLE":1}`，随后设备断开

## 7. 固件侧实现要点

### 7.1 模块入口
主程序调用：`blufi_custom_init()`（名称保留，但功能为 BLE JSON Provisioning）

### 7.2 NVS 存储
通过 `app_storage` 保存：
- `mode`（0=Wi‑Fi，1=4G）
- `ssid/password`（Wi‑Fi 模式）
- `mqtt_host/mqtt_port/full_url/username/password_mqtt`
- `pending_init`（用于 MQTT init 上报策略）

### 7.3 与网络管理器交互
收到 `statusNet` 后触发：
- `app_events_post_net_mode_request(mode)`

### 7.4 与 MQTT 管理器交互
收到 MQTT 配置后触发：
- `app_events_post_mqtt_config_updated()`
- 并进入 WAIT_BLE_CLOSE 等待 MQTT 状态回包

---

## 8. 日志与调试

固件串口输出关键日志（用于排障）：
1. 每次收到 JSON：`RX JSON (len=..., step=..., net_mode=...): ...`
2. 每次步骤迁移：`STEP XXX -> YYY (reason)`
3. 每次准备回包：`TX JSON: {...}`（即使 notify 未订阅也会打印）
4. 若 notify 未开启：`TX dropped: notify not enabled ...`
5. 步骤不匹配：`step mismatch ...`

---

## 9. 失败与边界条件

1. 未订阅 TX Notify：小程序收不到回包；串口会提示 `TX dropped: notify not enabled`。
2. 未按步骤发送：设备忽略并提示 `step mismatch`。
3. MQTT JSON 中含 `password` 字段：不会再被误判成 Wi‑Fi 配置（Wi‑Fi 配置必须含 `ssid`）。
4. 配网窗口超时：设备停止广播、断开连接，不再接受写入。

---

## 10. 验收清单（建议）
1. 上电后 120 秒内可扫描到 `WATER_PROV`；120 秒后不可扫描/不可写入。
2. 小程序按步骤发送，能收到：
   - `statusBLE` 回包
   - `statusNet` 回包
   - `statusWiFi` 0/1/2
   - `statusMQTT` 0/1
3. 乱序/重复发送不应导致设备进入错误状态（只应 step mismatch 并忽略）。
4. 单连接：第二个手机尝试连接应失败或排队（取决于系统行为），不会影响当前会话。
